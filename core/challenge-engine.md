# Challenge Engine

## Purpose
要求后续阶段对前序产物做最小必要的质疑与校验，避免盲信上游结论，及时发现缺口、偏差和更简单方案。

## Required Challenge Dimensions
每个后继阶段开始前，至少检查以下四类问题：
1. **完整性**：上游产物是否缺关键字段、章节或边界说明。
2. **可执行性**：上游产物是否足以支撑当前阶段继续工作。
3. **一致性**：当前阶段的输入、历史知识、已有实现与上游产物是否矛盾。
4. **简化空间**：是否存在更简单、更直接、更低风险的路径。

## Challenge Checklist
后续阶段在决定继续之前，至少检查以下四类问题：
1. 上游产物是否存在关键字段缺失。
2. 上游阶段是否留下了足够的读取证明与结论证据。
3. 上游阶段的边界判断是否与当前发现冲突。
4. 当前阶段是否因为上游问题而无法安全继续。

若任一项答案为“是”，应优先触发严格打回，而不是带着不合理内容继续推进。

- 任务拆分阶段应质疑实施方案是否可被原子化拆分。
- 编码阶段应质疑任务清单是否粒度合适、验证方式是否客观，并检查当前任务是否真的形成最小验证闭环。
- 审查阶段应质疑实现是否偏离任务、方案和需求，并检查是否过度设计，且不得在未搜证时直接下完成结论。
- 复盘阶段应质疑问题究竟来自执行疏漏、流程缺陷还是知识不足。

## Allowed Outcomes
质疑后只能有三种结果：
- 可以继续当前阶段。
- 需要补当前阶段产物。
- 必须回退到上游阶段。

## Rollback Output
若需要回退，必须明确：
- 回退阶段
- 回退原因
- 支持该判断的证据
- 上游需要补充的内容

## Result-Oriented Guidance
- 质疑的目的是提升结果质量，不是机械找茬。
- 若上游产物已足够支撑当前阶段，不需要人为制造问题。
- 若发现更简单方案，优先说明其价值，再判断是否需要回退重做。
