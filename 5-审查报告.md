# 工作流 5：审查报告

## 必须加载 (Mandatory Core Rules)
- `core/workflow-engine.md`
- `core/stage-gates.md`
- `core/evidence-engine.md`
- `core/challenge-engine.md`
- `core/memory-engine.md`
- `core/simplification-engine.md`
- `references/glossary.md`
- `references/frontend-standards.md` 或 `references/backend-standards.md`
- `stages/5-review.md`
- `C:\knowledge\INDEX.md`
- `当前项目 docs/architecture/INDEX.md（若存在）`
- **MCP Tool: sequential-thinking** (必须使用该工具进行增量Diff审查、一致性核对与视觉/性能指标评估)

## 核心规程 (Core Rule)
本工作流运行 `Review` 阶段。目标是：基于增量改动检查实现是否满足需求、方案与任务，并识别是否存在更简单、更稳的实现方式。

## 共享知识库公约 (Shared Knowledge Location)
- 7 个工作流共用一套单层跨项目共享知识库：`C:\knowledge`。
- `C:\knowledge` 只保存跨项目长期知识，例如协作规则、编码偏好、设计偏好、长期经验、流程偏好。
- 项目结构、模块边界、接口契约、业务规则、项目技术决策必须维护在当前项目目录内，例如 `docs/architecture/`，不要写入 `C:\knowledge`。
- 禁止在 `C:\knowledge` 中创建 `global/`、`project/`、`global_workflows/` 目录或 core/stages 等冗余副本。

## 硬门禁 (Hard Gates)
- 先执行真实物理文件读取，再做阶段判断；禁止只凭缓存、记忆或口头声明“已加载”。
- 读取必须覆盖本工作流列出的全部 Mandatory Physical Reads；必须在阶段产物中写入 `Mandatory Reads` 与 `Read Evidence`。
- **思考工具硬门禁 (Sequential Thinking Hard Gate)**: 在评估代码一致性、编写审查报告前，**必须**调用 `sequential-thinking` 工具进行不少于5个 thought 步骤的审查推导，逐项核对增量代码是否偏离 Requirement、Design 或 Task。如果是前端任务，**必须**使用该工具系统评估界面交互流畅度与动画设计是否符合 premium 体验标准。阶段产物中**必须**记录思考调用证据和推演结论（`Logic Thinking Evidence`），无此项证据禁止通过此阶段。
- 阶段产物必须写成可回读物理文件；禁止只在对话框中输出产物。
- 缺少上游产物、必读知识文件未读、Read Evidence 缺失、阶段产物未落盘、关键字段缺失时，停止推进并写明 Blockers 与 Rollback Advice.
- 任何“已完成、无问题、可进入下一阶段、已闭环”的结论都必须有证据；无证据视为未完成。
- 如果阶段协议要求维护知识库或本轮产生了可复用知识，必须实际更新 `C:\knowledge` 中的跨项目知识，或更新当前项目内的架构/项目知识文档，并在产物中写入 `Knowledge Update Evidence`。

## 必须物理读取 (Mandatory Physical Reads)
- `C:\knowledge\INDEX.md`
- 当前项目 `docs/architecture/INDEX.md`（若审查涉及项目结构一致性且该文件存在）

## 必读规程 (Required Reading Procedure)
1. 读取当前工作流入口中的必须加载文件列表、Mandatory Physical Reads 与各 Gate。
2. 物理读取全部 Mandatory Physical Reads。
3. **必须启动 sequential-thinking**：使用 `sequential-thinking` MCP 工具对增量 Diff 与任务清单、方案规范进行多轮推演比对，挑剔地寻找逻辑漏洞、视觉设计死角与潜在重构空间。
4. 按阶段协议中声明 of 产物结构创建或更新阶段产物。
5. 读取必要的上游阶段产物（`.workflow/task-list.md`、需求、方案等）与增量改动。
6. 物理回读对应阶段的 Artifact 物理文件，禁止凭借记忆“脑补”审查。
7. 读取 `C:\knowledge` 时先读 `C:\knowledge\INDEX.md`，再按相关性读取正文片段。

## 常见阻断违规 (Common Violations to Block)
- 未实际调用 `sequential-thinking` 思考工具分析 Diff 代码的一致性、健壮性与视觉交互细节。
- 口头宣称“已读”但没有工具读取证据。
- 只在聊天对话中输出，不创建或更新 `.workflow/review-report.md` 物理文件。
- 阶段产物中缺失 `Logic Thinking Evidence` 或 `Read Evidence`。
- 上游产物缺失仍继续下阶段。
- 未做证据检索就直接写“无问题”“已完成”“已闭环”等结论。
- 凭记忆脑补审查，未发生真实的物理动作回读产物文件。

## 阶段门禁细则 (Gates)
- **Input Gate**：必须存在本次增量改动与 `.workflow/task-list.md` 或等价 Task List。若缺少任务上下文，至少应补齐当前改动范围与对应任务目标。
- **Scope Gate**：禁止借审查名义新增无关功能或大范围重构。禁止未读取 Requirement、Plan、Task 与增量改动就写一致性结论。禁止无证据宣称无问题、已闭环、可上线。必须包含 `Logic Thinking Evidence`。
- **Artifact Gate**：必须创建或更新 `.workflow/review-report.md`；必须包含 Findings、Final Assessment、Blockers、Rollback Advice、Completion Conditions。
- **Knowledge Gate**：跨项目审查经验只写入 `C:\knowledge`；项目架构认知必须维护在当前项目 `docs/architecture/`。
- **Exit Gate**：若 Review Report 已形成且重要问题已处理，且通过了 Sequential Thinking Hard Gate，可结束本轮业务流或下一步建议为 `/6-复盘优化`。若门禁不满足，必须给出明确 Blockers 与 Rollback Advice。
