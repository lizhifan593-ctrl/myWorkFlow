# 工作流 1：需求确认

## 必须加载 (Mandatory Core Rules)
- `core/workflow-engine.md`
- `core/stage-gates.md`
- `core/evidence-engine.md`
- `core/memory-engine.md`
- `core/challenge-engine.md`
- `core/simplification-engine.md`
- `references/glossary.md`
- `stages/1-requirement.md`
- `C:\knowledge\INDEX.md`
- **MCP Tool: sequential-thinking** (必须使用该工具对需求进行推演与拆解)

## 核心规程 (Core Rule)
本工作流运行 `Requirement` 阶段。目标是：将模糊需求收敛为可执行、可验证、可交接的 `Requirement Contract`。

## 共享知识库公约 (Shared Knowledge Location)
- 7 个工作流共用一套单层跨项目共享知识库：`C:\knowledge`。
- `C:\knowledge` 只保存跨项目长期知识，例如协作规则、编码偏好、设计偏好、长期经验、流程偏好。
- 项目结构、模块边界、接口契约、业务规则、项目技术决策必须维护在当前项目目录内，例如 `docs/architecture/`，不要写入 `C:\knowledge`。
- 禁止在 `C:\knowledge` 中创建 `global/`、`project/`、`global_workflows/` 目录或 core/stages 等冗余副本。

## 硬门禁 (Hard Gates)
- 先执行真实物理文件读取（如调用 `view_file`），再做阶段判断；禁止只凭缓存、记忆或口头声明“已加载”。
- 读取必须覆盖本工作流列出的全部 Mandatory Physical Reads；必须在阶段产物中写入 `Mandatory Reads` 与 `Read Evidence`。
- **关系图谱硬门禁 (CodeGraph Hard Gate)**：每次进入需求确认阶段前，**必须**首先检查项目目录下是否存在关系图谱（即是否存在 `.codegraph/codegraph.db` 文件）。若不存在，则**必须**在开始后续工作前运行 `npx @colbymchenry/codegraph init` 建立并索引本地关系图谱，并在阶段产物（Requirement Contract）中记录关系图谱验证证据（`CodeGraph Verification Evidence`）。
- **思考工具硬门禁 (Sequential Thinking Hard Gate)**: 在分析需求与澄清模糊点前，**必须**调用 `sequential-thinking` 工具进行针对性的多步骤逻辑推导（不得少于5个 thought 步骤），分析目标、核心逻辑与系统边界。在阶段产物中**必须**记录思考调用证据和推演结论（`Logic Thinking Evidence`），无此项证据禁止通过此阶段。
- 阶段产物必须写成可回读物理文件；禁止只在对话框中输出产物。
- 缺少上游产物、必读知识文件未读、Read Evidence 缺失、阶段产物未落盘、关键字段缺失时，停止推进并写明 Blockers 与 Rollback Advice。
- 任何“已完成、无问题、可进入下一阶段、已闭环”的结论都必须有证据；无证据视为未完成。
- 如果阶段协议要求维护知识库或本轮产生了可复用知识，必须实际更新 `C:\knowledge` 中的跨项目知识，或更新当前项目内的架构/项目知识文档，并在产物中写入 `Knowledge Update Evidence`。

## 必须物理读取 (Mandatory Physical Reads)
- `C:\knowledge\INDEX.md`
- 若项目存在本地架构索引，优先读取 `docs/architecture/INDEX.md`

## 必读规程 (Required Reading Procedure)
1. 读取当前工作流入口中的必须加载文件列表、Mandatory Physical Reads 与各 Gate。
2. 检查项目根目录下是否存在关系图谱（`.codegraph/codegraph.db`）。若没有，先运行 `npx @colbymchenry/codegraph init` 创建并建立关系图谱。
3. 物理读取全部 Mandatory Physical Reads。
4. **必须启动 sequential-thinking**：使用 `sequential-thinking` MCP 工具对用户原始需求进行深度剖析，挖掘深层假设，盘点受影响的现有模块，理清接口边界，并记录详细逻辑推演步骤。
5. 按阶段协议中声明的产物结构创建或更新阶段产物。
6. 读取必要的上游阶段产物（若有）。
7. 优先物理回读项目本地架构入口 `docs/architecture/INDEX.md` 以防踩坑。
8. 读取 `C:\knowledge` 时先读 `C:\knowledge\INDEX.md`，再按相关性读取正文片段。

## 常见阻断违规 (Common Violations to Block)
- 未实际调用 `sequential-thinking` 工具进行深度逻辑思考推演，或者调用思考步骤过少（如仅1步或敷衍了事）。
- 口头宣称“已读”但没有工具读取证据。
- 只在聊天对话中输出 Requirement Contract，不创建或更新 `.workflow/` 下的阶段物理文件。
- 阶段产物中缺失 `Logic Thinking Evidence` 或 `Read Evidence`。
- 上游产物缺失仍继续下阶段。
- 只打勾不写任务 Notes、证据、阻塞和回退条件。
- 复盘只写反思但没有实际规则或知识文件更新。
- 把项目具体业务架构写进 `C:\knowledge`。

## 阶段门禁细则 (Gates)
- **Input Gate**：允许从空上下文开始。若存在更高阶段历史产物，本次进入视为重新启动需求确认，以新目标重新收敛。
- **Scope Gate**：禁止写实现代码、技术方案、任务清单或交付报告。禁止只在对话中输出，必须写入物理文件 `.workflow/requirement-contract.md`。必须包含 `Logic Thinking Evidence`。
- **Artifact Gate**：必须创建或更新 `.workflow/requirement-contract.md`（遵循 `templates/requirement-contract.md` 结构）。
- **Knowledge Gate**：必须先读 `C:\knowledge\INDEX.md`。若形成跨项目长期偏好、协作规则，更新并记录 `Knowledge Update Evidence`。更新知识前必须优先依据可物理回读的阶段产物与最终结果。
- **Exit Gate**：若 Requirement Contract 已形成且关键模糊点已关闭，且通过了 Sequential Thinking Hard Gate，唯一下一步建议为 `/2-方案设计`。若任一硬门禁不满足，不得推荐下一阶段，必须给出明确 Blockers 与 Rollback Advice。
