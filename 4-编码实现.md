## 核心规程 (Core Rule)

## 共享知识库公约 (Shared Knowledge Location)
- 7 个工作流共用一套单层跨项目共享知识库:`C:\knowledge`。
- `C:\knowledge` 只保存跨项目长期知识(协作规则、编码偏好、设计偏好、长期经验、流程偏好)。
- 项目结构、模块边界、接口契约、业务规则、项目技术决策必须维护在当前项目目录内(如 `docs/architecture/`),不写入 `C:\knowledge`。
- 禁止在 `C:\knowledge` 中创建 `global/` / `project/` / `global_workflows/` 目录或 core/stages 等冗余副本。

## 必读规程 (Required Reading Procedure — 固定 7 步)
> 本节定义本入口必须触发的物理读取动作。门禁细则与禁止行为见 `core/stage-gates.md` 与 `stages/4-implementation.md`,本入口不重抄。

1. **加载规则**:物理读取本入口列出的"必须加载"中所有 `core/` / `references/` / `stages/` / `templates/` 文件,登记到 Evidence Ledger。
2. **回读上游产物**:物理读取 `.workflow/task-list.md`,锁定**当前一个 `T-id`**,登记 `E-id`;若缺失或粒度不合理,触发 Rollback to /3-任务拆分。
3. **验收输入**:在 Task List 当前任务的 `Stage Evidence` / `Notes` 中标注 `Traces`(`M-id`, `AC-id`)与 `Source Diff Targets`,确认与上游一致。
4. **探索源码、图谱、知识**:按 `stages/4-implementation.md` 与 `core/codegraph-engine.md` 中本阶段最少动作清单执行 codegraph 查询(改前 `find_callers` 必须执行;新增对外符号后 `search_symbol` 自检);若任务涉及核心业务链路,物理读取项目本地 `docs/architecture/INDEX.md` 与对应子模块文档,登记 `E-id`。
5. **产出文件**:精准修改与当前任务相关的代码;更新 `.workflow/task-list.md` 的当前 `T-id`,在 Notes 中追加 codegraph 调用记录(`E-id`)与关键逻辑变化、踩坑、临时妥协;Stage Evidence 至少包含 1 条 `Type=Command` 的真实编译/测试/运行证据。
6. **回读自检**:执行 `core/anti-hallucination.md` 中的 Verification Order;核对 Claim Evidence Map 中"已完成 / 已验证"必须挂高置信 `E-id`;工具失败时只能写 `Blocked / Not verified / Pending`。
7. **下一步建议**:当前任务完成且证据齐备则继续下一个 `T-id` 或推荐 `/5-审查报告`;否则只写 `Blockers` + `Rollback Advice`。
