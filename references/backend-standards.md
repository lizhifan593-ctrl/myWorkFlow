# 后端架构与数据契约基准 (Backend Standards)

## 1. 三层严格解耦架构 (The 3-Tier Rule)
- **Router (路由接入层)**：仅负责 HTTP 请求验证、参数绑定解析、依赖注入。**严禁**在这一层编写任何具体业务流转逻辑或重组数据形态。
- **Service (核心服务层)**：承载所有的“业务规则”、状态演进、复杂多表聚合计算。所有关于通知下发、外部调用和数据一致性比对的步骤必须置于此层。
- **DAO / CRUD (数据访问层)**：极其纯粹的入库/出库层。仅包含 ORM（如 SQLAlchemy / Hibernate）原生的查增删改语法，**绝对禁止**手动进行字典组装或数据结构强行拼接。

## 2. 自动化序列化与数据契约
- **零侵入契约映射**：彻底废弃在代码中手动拼接与组装返回海量 JSON 的落后做法。必须依靠 ORM 与传输模型（DTO/Schema）之间的自动化关系转换（例如 Pydantic 的 `from_attributes=True` 或 ORM `@property`），由框架底层接管数据清洗。
- **模型生命周期对齐防漏改**：一旦数据库 Model 发生变更（特别是新增了必填约束字段），开发人员必须进行地毯式检索并同步该实体的所有前后端传输结构（Create、Update、Response 模型），防止发生因入参验证不全导致的静默更新失效或内部 500 异常。

## 3. 防坑与安全机制
- **排序防崩溃范式**：在跨库业务场景下，查询 `ORDER BY` 中严禁使用针对特定方言的“空值置底”语法（如 `.nullslast()` 极易造成特定数据库崩溃抛错）。强制使用通用范式：`Model.field.is_(None)` 做首要排序位以实现安全跨库的兼容兜底。
- **OOM (内存打满) 拦截机制**：一切涉及大文件读取与下载的接口，绝对禁止使用无上限的全量二进制读取方法（`read()`）。必须硬性设定流式读取或分块上限，数据大小发生溢出时需立即熔断拦截请求，防止服务雪崩。
- **路由遮蔽拦截**：所有特定的静态操作路由入口（如 `/api/resource/export`）必须在代码定义层面排在动态参数路由（如 `/api/resource/{id}`）之上，否则将导致静态路径被动态参数强制接管转换并引发无法预估的解析报错。

## 4. 容灾与时间线审计
- **原子日志落盘**：对由用户核心操作引发的业务数据的变更，审计流水必须精确到原子操作粒度。修改单一属性生成独立日志片段，禁止为了省事打包冗余信息，影响下游界面的精细化时间线渲染。
- **防堆积数据隔离 (Anti-bloat)**：为了确保持久化服务的磁盘和内存健康，任何上传的非绑定态附件或临时结构应首先定义为“孤儿态”，并部署周期性后台巡检协程/定时任务，对过期或异常遗留脏数据执行坚决的物理清除。
