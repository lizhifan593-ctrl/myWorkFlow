# Knowledge Quality Standards

## Purpose
定义什么内容值得进入知识库、如何写、如何更新、如何清理，确保知识库是高质量长期资产，而不是噪音仓库。

## Inclusion Standards
一条知识进入知识库前，至少满足以下条件中的两个：
- 未来多个任务中仍可能有用。
- 不容易直接从代码、目录或临时对话中重新获得。
- 会实质改善后续阶段的判断、边界控制或结果质量。

## Exclusion Standards
以下内容不应进入长期知识库：
- 一次性任务细节。
- 临时执行状态。
- 纯调试过程碎片。
- 纯情绪化结论。
- 很快会失效且无法复用的信息。

## Writing Standards
每条知识尽量回答四个问题：
- 这是什么事实或偏好？
- 为什么重要？
- 该在什么场景使用？
- 可信度如何？

## Update Standards
- 能更新原文件时，优先更新，不无节制新增新文件。
- 新证据与旧知识冲突时，优先修订旧知识。
- 过时知识比没有知识更危险，应及时修订或移除。

## Knowledge Update Evidence Rule
- 知识维护优先基于可回读的阶段产物与最终结果,而不是依赖长对话上下文。
- 若对话上下文已被压缩,知识总结应优先回读相关阶段产物,再回读最终结果。
- 若阶段产物与实际结论不一致,应先修正阶段产物,再写知识库。
- 知识与技能条目如来源于阶段产物,**必须**在 Knowledge Update Evidence 中按 ID 或路径引用:
  - 来自 Design Contract 的需求、设计偏好或约束 → 引用 `AC-id` / `M-id` / `A-id`。
  - 来自 Task List 的实施拓扑与踩坑 → 引用 `T-id`。
  - 来自 Review Report 的缺陷模式与验证 Trace → 引用 `F-id`。
  - 来自 Retro & Evolution Report 的规则修改或自编译技能 → 引用 `Change-id` 或 `SKILL.md` 物理路径。
- 知识条目可以同时挂多个 ID;但每条至少要挂 1 个 `E-id`(来自对应阶段的 Evidence Ledger),否则视为伪证据。

## Reading Standards
- 知识读取遵循:索引 → 摘要 → 正文片段。
- 禁止默认全量读取整个知识库。
- 只有当知识确实影响当前阶段判断时,才深入读取正文。
- 每次读取必须在阶段产物的 Evidence Ledger 登记 `Type=Read` 条目;未登记的"已读"声明视为伪证据。

## Usage Standards
- 知识库不只是记录,还必须能影响阶段产物。
- 在关键产物中应尽量显式写出 `Knowledge Applied`,说明知识的使用影响,并在指向具体决策时引用对应 ID(`AC-id` / `M-id` / `T-id` / `F-id`)。
- 若知识未影响当前判断,不应为了形式而硬写引用。
